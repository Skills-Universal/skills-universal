# SKILL: Algorithm Source Intelligence\n*Pattern per costruire algoritmi che si migliorano continuamente da fonti esterne e dal campo.*\n\n---\n\n## Quando usare questa skill\n\nOgni volta che si sviluppa o mantiene un **algoritmo domain-specific** che:\n- Ha basi scientifiche/fisiche (meteo, finanza, logistica, medicina...)\n- Pu\u00f2 essere confrontato con implementazioni open source di alta qualit\u00e0\n- Produce output verificabili contro risultati reali\n- Evolve con nuove conoscenze del dominio\n\n---\n\n## Il pattern in 4 componenti\n\n### 1. Indice delle fonti (`SOURCES.md`)\n\nFile vivo nel repo che cataloga **tutte le fonti studiate**, adottate o scartate.\n\n**Struttura minima per ogni fonte:**\n```markdown\n### `owner/repo` ⭐⭐⭐⭐\n- **Cosa fa**: descrizione in una riga\n- **Qualità algoritmica**: Alta/Media/Bassa — motivazione\n- **Rilevanza operativa**: Alta/Media/Bassa — perché sì/no\n- **Stato**: Attivo/Archiviato (anno ultimo commit)\n- **Cosa abbiamo adottato**: lista specifica di funzioni/pattern\n- **Cosa resta da valutare**: backlog esplicito\n- **File chiave**: percorsi specifici da leggere\n```\n\n**Regola critica**: scartare una fonte è informazione preziosa quanto adottarla. Documentare il motivo dello scarto evita di rivalutare le stesse fonti in sessioni future.\n\n---\n\n### 2. Script di discovery (`source_scout.py`)\n\nScript autonomo che cerca nuovi repo di qualità e propone aggiornamenti.\n\n**Pattern implementativo:**\n```python\nSEARCH_QUERIES = [...]      # query GitHub Search API\nKNOWN_REPOS = {...}          # repo già valutati — non riproporre\nRELEVANCE_KEYWORDS = [...]   # parole chiave del dominio\nMIN_STARS = N                # soglia minima qualità\n\ndef relevance_score(repo) -> float:\n    # Combina: keyword match + stelle + manutenzione recente\n    # Malus: archiviato, fork\n    ...\n```\n\n**Output**: report stdout + aggiornamento opzionale SOURCES.md con sezione `Nuove fonti candidate`.\n\n**Frequenza consigliata**: eseguire ogni 30-60 giorni o prima di ogni sprint algoritmico.\n\n---\n\n### 3. Sezione \"Lezioni dal campo\" in SOURCES.md\n\nDopo ogni evento reale confrontabile con le previsioni dell'algoritmo:\n\n```markdown\n### Evento: [descrizione] — [data]\n- **Previsione algoritmo**: [output]\n- **Esito reale**: [cosa è successo]\n- **Valutazione**: Corretta/Parziale/Errata — perché\n- **Modello/fonte più accurata**: [quale fonte/modello ha performato meglio]\n- **Apprendimento**: [cosa cambia nell'algoritmo]\n- **Fix applicato**: [link commit o \"da fare\"]\n```\n\n**Regola**: non aspettare la sessione di sviluppo successiva. Aggiornare subito dopo l'evento mentre i dettagli sono freschi.\n\n---\n\n### 4. Valutazione prima di adottare\n\nPrima di integrare codice da una fonte esterna, rispondere a queste domande:\n\n1. **Il metodo è fisicamente motivato o è una euristica empirica?** (preferire fisica motivata)\n2. **Richiede dati che già fetchiamo?** (se no, stimare costo di aggiungere il fetcher)\n3. **Ha test che possiamo eseguire?** (verificare sempre con i nostri dati reali)\n4. **Cambia l'output in modo verificabile?** (se non possiamo misurare il miglioramento, non adottare)\n5. **È retrocompatibile?** (aggiungere parametri opzionali, mai rompere interfacce esistenti)\n\n---\n\n## Processo di adozione\n\n```\n1. Leggere il file chiave nella fonte (non il README)\n2. Isolare la funzione/pattern specifico\n3. Scrivere test con dati reali storici\n4. Confrontare output vecchio vs nuovo\n5. Se miglioramento misurabile → integrare\n6. Aggiornare SOURCES.md (spostare da \"Da valutare\" ad \"Adottato\")\n7. Documentare il fix nel changelog/commit con riferimento alla fonte\n```\n\n---\n\n## Anti-pattern da evitare\n\n- **Adottare senza testare**: il codice di qualità di altri potrebbe non funzionare nel tuo contesto\n- **Copiare senza capire**: capire ogni riga prima di integrare — non importare black box\n- **Rivalutare fonti già scartate**: SOURCES.md serve esattamente per evitare questo\n- **Accumulare \"da valutare\" senza priorità**: ogni sprint deve avere max 2-3 item in evaluation\n- **Ignorare le lezioni dal campo**: i dati reali sono più preziosi di qualsiasi repo\n\n---\n\n## Integrazione con il ciclo di sviluppo\n\n```\nInizio sessione algoritmica:\n  1. Leggere SOURCES.md — sezione \"Metriche mancanti\"\n  2. Scegliere max 1-2 item da implementare\n  3. Leggere il file chiave nella fonte di ispirazione\n  4. Implementare con test\n\nFine sessione:\n  5. Aggiornare SOURCES.md — spostare item implementati\n  6. Aggiungere nuovi \"Da valutare\" se emersi\n\nOgni 30-60 giorni:\n  7. Eseguire source_scout.py\n  8. Aggiungere nuove fonti rilevanti a SOURCES.md\n  9. Rivedere \"Lezioni dal campo\" e aggiornare soglie/pesi\n```\n\n---\n\n## Esempio: AstroWeather Engine\n\nImplementazione di riferimento:\n- `fabrizio-marra/astroweather` \u2192 `docs/SOURCES.md` \u2014 indice fonti\n- `fabrizio-marra/astroweather` \u2192 `scripts/source_scout.py` \u2014 discovery automatico\n\nLezione concreta: il bug `_phase_name` (luna piena classificata come \"ultimo quarto\") è stato trovato confrontando i nostri dati reali (illuminazione 98.8%) con il metodo `lunation` di `mawinkler/pyastroweatherio`. Senza l'indice delle fonti non avremmo saputo dove cercare.\n\n---\n\n*Skill creata: 2026-05-01 — AstroWeather Engine sprint*\n*Applicabile a: qualsiasi algoritmo domain-specific con output verificabile*\n
